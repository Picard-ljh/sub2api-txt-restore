# sub2api TXT 转 JSON 离线工具

这是一个专门用来把“一行一号”的 TXT 卡密内容转换成 sub2api 可导入 JSON 的本地离线工具。

它适合这种场景：

1. 卖家从 sub2api 导出账号 JSON。
2. 卖家把账号拆成一行一个账号的 TXT 卡密。
3. 链动小铺等平台按“一行一个卡密”的形式发给买家。
4. 买家拿到多行卡密后，不能直接导入 sub2api。
5. 买家使用本工具把多行卡密还原成一个标准 sub2api JSON 文件。
6. 买家进入 sub2api 后台，把生成的 JSON 文件导入。

本工具不提供在线服务，也不需要服务器。下载 HTML 文件后，直接在本地浏览器打开即可使用。

## 这个工具会不会上传我的账号数据？

不会。

本工具是一个纯前端离线 HTML 文件。文件读取、内容解析、JSON 生成、文件下载都在你自己的浏览器里完成。

它不会：

- 上传账号数据
- 请求服务器
- 保存账号到数据库
- 保存账号到浏览器 localStorage
- 接入统计脚本
- 接入广告或第三方追踪

如果你仍然不放心，可以先断开网络，再双击打开 HTML 文件使用。

## 最简单的使用方法

1. 打开本仓库的 Releases 页面。
2. 下载 `sub2api-txt-to-json-offline.html`。
3. 下载完成后，双击这个 HTML 文件。
4. 浏览器会打开一个本地页面。
5. 把你购买到的卡密全部复制进去，一行一个。比如买了 80 个号，就粘贴 80 行。
6. 粘贴完成后，先点击页面底部的“检查内容”按钮。
7. 右侧会显示检查结果。请确认“账号数”和你购买的数量一致，并且“错误”为 0。
8. 只有检查通过后，“下载 sub2api JSON”按钮才可以点击。
9. 点击“下载 sub2api JSON”。
10. 浏览器会下载一个 `sub2api-account-import_时间.json` 文件。
11. 打开你的 sub2api 后台。
12. 进入账号管理页面。
13. 使用“数据导入”功能导入刚刚下载的 JSON 文件。

简单说就是：

```txt
粘贴卡密 -> 点击“检查内容” -> 确认账号数正确 -> 点击“下载 sub2api JSON” -> 去 sub2api 后台导入
```

如果没有先点击“检查内容”，下载按钮会保持不可用，这是正常的。

注意：不要在 GitHub 的代码预览页面里直接复制 HTML 源码使用。普通用户只需要从 Releases 下载 `sub2api-txt-to-json-offline.html`，然后本地双击打开。

## 输入格式是什么？

输入是“一行一个账号”的 TXT 内容。每一行通常长这样：

```txt
{"accounts":[{"name":"账号1","platform":"openai","type":"oauth","credentials":{"access_token":"示例"}}],"proxies":[]}
{"accounts":[{"name":"账号2","platform":"openai","type":"oauth","credentials":{"access_token":"示例"}}],"proxies":[]}
```

如果你购买了 80 个号，通常就会得到 80 行。你可以一次性全部粘贴进去。

## 输出格式是什么？

工具会生成一个 `.json` 文件，结构类似：

```json
{
  "exported_at": "2026-06-20T00:00:00.000Z",
  "proxies": [],
  "accounts": [
    {
      "name": "账号1",
      "platform": "openai",
      "type": "oauth",
      "credentials": {}
    },
    {
      "name": "账号2",
      "platform": "openai",
      "type": "oauth",
      "credentials": {}
    }
  ]
}
```

也就是说：

- 粘贴 1 行，输出 1 个账号的 JSON。
- 粘贴 80 行，输出 1 个 JSON 文件，里面有 80 个账号。
- 粘贴 100 行，输出 1 个 JSON 文件，里面有 100 个账号。

它不会把 80 行导出成 80 个文件。

## 为什么不能直接把 TXT 导入 sub2api？

因为 sub2api 后台导入需要的是一个标准 JSON 文件，而链动小铺等平台发放卡密时通常是一行一个账号。

一行一号的 TXT 更适合发货，但不适合直接导入 sub2api。

本工具的作用就是把多行卡密合并回一个 sub2api 可以识别的 JSON 文件。

## 如何确认转换结果对不对？

粘贴卡密后，先点击“检查内容”。点击以后，你可以看工具页面右侧的检查结果：

- 有效行：成功识别了多少行卡密
- 账号数：最终会导出多少个账号
- 错误：有多少行无法识别
- 代理：是否包含代理信息

正常情况下，如果你购买了 80 个号，右侧“账号数”应该显示 80。

如果显示的账号数量和你购买的数量不一致，不要急着导入。先检查是不是少复制了某一行，或者某一行内容被聊天软件折断了。

如果右侧“错误”不是 0，也不要下载导入。先根据错误提示修正对应行，再重新点击“检查内容”。

## 出现错误怎么办？

常见错误有几种：

### 提示“这一行不是合法 JSON”

说明这一行卡密可能没有复制完整，或者被多复制了额外字符。

处理方法：

1. 找到提示里的行号。
2. 回到输入框检查对应行。
3. 确认这一行是否从 `{` 开始，并以 `}` 结束。
4. 重新复制完整卡密。

### 提示“缺少 accounts 数组”

说明这一行不是本工具需要的一行一号格式。

正确的一行通常应该包含：

```txt
"accounts":[...]
```

### 提示账号数量不对

可能原因：

- 少复制了某些卡密
- 多复制了空白说明文字
- 有一行卡密被换行拆断
- 有重复卡密

建议重新从购买页面复制完整卡密内容，再检查一次。

## 导入 sub2api 前建议做什么？

建议先备份你当前 sub2api 里的账号数据。

原因是账号导入属于数据写入操作。即使本工具只是生成导入文件，也建议你在真正导入前保留一份原始备份，方便出现误操作时恢复。

## 这个工具会不会修改 token？

不会。

本工具只做三件事：

1. 逐行解析 TXT。
2. 提取每一行里的 accounts。
3. 合并成一个标准 JSON。

它不会刷新 token，不会重写 token，不会登录你的账号，也不会联系 sub2api 服务端。

## 这个工具会不会删除重复账号？

不会自动删除。

如果检测到重复账号名或重复邮箱，工具会提示你，但不会擅自删除任何账号。因为某些情况下，重复名称可能是卖家本来就这样设置的。

## 可以把真实卡密发到 GitHub Issue 里求助吗？

不要。

如果你要反馈问题，请先把账号里的敏感字段删掉或替换成 `fake-token`。

不要公开发布这些内容：

- access_token
- refresh_token
- id_token
- 邮箱完整地址
- 真实账号 JSON
- 真实卡密 TXT

## 开发者说明

本项目使用 Vite、TypeScript 和 Vitest。

安装依赖：

```bash
npm install
```

本地开发：

```bash
npm run dev
```

运行测试：

```bash
npm test
```

构建普通静态页面：

```bash
npm run build
```

构建单文件离线版：

```bash
npm run build:offline
```

构建完成后，成品会生成在：

```txt
release/sub2api-txt-to-json-offline.html
```

同时会生成：

```txt
release/SHA256SUMS.txt
```

用于校验 HTML 文件是否被篡改。

## 发布版本建议

维护者发布新版本时，建议在 GitHub Release 里上传两个文件：

```txt
sub2api-txt-to-json-offline.html
SHA256SUMS.txt
```

Release 说明里建议写清楚：

```txt
这是离线本地工具。下载 HTML 文件后，用浏览器打开即可使用。请不要把真实账号、真实卡密或 token 发到 Issue。
```
